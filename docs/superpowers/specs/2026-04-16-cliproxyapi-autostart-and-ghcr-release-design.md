# CLIProxyAPI Auto-Start and GHCR Release Automation Design

## Summary

This design combines two focused operational improvements:

1. Harden `cliproxyapi` Docker lifecycle behavior so repo-managed flows are less likely to leave the service stopped or missing, while preserving Docker-native auto-restart behavior.
2. Make the dashboard release workflow automatically publish the dashboard container image to GHCR instead of relying on a manual-only release workflow.

The scope is intentionally narrow: preserve the current architecture, keep GHCR as the only registry target, and improve reliability around the existing Docker and GitHub Actions paths rather than redesigning deployment from scratch.

---

## Goals

- Keep `cliproxyapi` aligned with Docker-native auto-start behavior.
- Reduce repo-defined flows that can strand `cliproxyapi` in a stopped or removed state after update/rebuild failures.
- Preserve the current operational model where non-Docker pieces may still exist.
- Automatically publish the dashboard image to GHCR when the release/tag path runs.
- Keep the current dashboard image naming convention and tagging model.

## Non-Goals

- Remove all host-level dependencies such as systemd, cron, or host proxy integrations.
- Guarantee zero downtime for every update or rebuild path.
- Prevent operators from manually stopping the container.
- Add Docker Hub or any registry besides GHCR.
- Replace the existing release system with a new CI/CD architecture.
- Change the dashboard update notification system in this work.

---

## Current State

### `cliproxyapi` lifecycle

Current repo behavior already includes Docker auto-restart semantics through:

- `infrastructure/docker-compose.yml`
- `docker-compose.local.yml`
- `docker-compose.yml`

where `cliproxyapi` uses `restart: unless-stopped`.

Current repo behavior can still stop/remove/recreate the container through:

- `dashboard/src/app/api/update/route.ts`
  - compose-based `up -d --no-deps --force-recreate`
  - fallback `docker rm -f cliproxyapi` and `docker run ...`
- `rebuild.sh`
  - default path uses `docker compose down` then `docker compose up -d`
- systemd service generated by `install.sh`
  - `ExecStop=/usr/bin/docker compose down`

So the problem is not “Docker cannot auto-start this container”; the problem is that some repo-managed flows intentionally remove/recreate or tear the stack down, and recovery expectations are not explicit enough.

### Dashboard GHCR publishing

Current release pipeline exists in:

- `.github/workflows/release.yml`

It already knows how to build and push the dashboard image to:

- `ghcr.io/${{ github.repository }}/dashboard`

But it is currently `workflow_dispatch` only, so normal release/tag flow does not automatically publish unless someone manually runs the workflow.

---

## Design Overview

The implementation will keep both changes small and targeted.

### Part A — `cliproxyapi` auto-start hardening

#### Core approach

- Keep `restart: unless-stopped` as the primary Docker restart behavior.
- Preserve systemd and other non-Docker runtime integrations as-is.
- Harden repo-managed lifecycle paths so their intended post-condition is always:
  - `cliproxyapi` is running again after the operation completes or after a bounded recovery path executes.

#### File boundaries

- `infrastructure/docker-compose.yml`
  - verify/retain the restart policy as the production source of truth.
- `docker-compose.local.yml` and `docker-compose.yml`
  - keep parity only if needed so local behavior does not diverge confusingly.
- `dashboard/src/app/api/update/route.ts`
  - make update/recovery behavior explicitly compose-first and recovery-aware.
- `rebuild.sh`
  - clarify and, if needed, harden the default rebuild path so the final state expectation is explicit.
- `docs/INSTALLATION.md`
- `docs/SERVICE-MANAGEMENT.md`
- optionally `docs/RUNBOOK.md`
  - document which flows preserve continuity vs deliberately tear down and recreate the stack.

#### Behavior rules

- Host reboot / Docker daemon restart:
  - `cliproxyapi` should come back via Docker restart policy and existing stack startup path.
- Repo-managed update path:
  - prefer compose-managed recreation/recovery over ad-hoc fallback when compose metadata is available.
- Rebuild path:
  - full-stack rebuild may still recreate the container, but docs and script behavior must make the intended post-condition clear.
- Manual operator stop/down:
  - still allowed; this design does not remove that capability.

#### Trade-off

This design improves “service comes back correctly” rather than “service can never be recreated.” That is acceptable because the user asked for auto-start/auto-restart, not zero-recreate semantics.

---

### Part B — automatic dashboard image publish to GHCR

#### Core approach

- Reuse the existing release workflow in `.github/workflows/release.yml`.
- Keep GHCR as the only target registry.
- Make the publish path automatic for the release/tag flow instead of manual-only.

#### File boundary

- `.github/workflows/release.yml`

#### Behavior rules

- Dashboard image continues to publish to:
  - `ghcr.io/<owner>/<repo>/dashboard`
- Published tags continue to include the current intended set:
  - release version
  - `latest`
  - `sha-<shortsha>` if retained by the existing workflow logic
- Workflow trigger is no longer limited to manual dispatch for the publish path.
- The design should prefer small trigger/condition changes over a full workflow split unless the existing job structure makes that impossible to reason about safely.

#### Trade-off

Keeping everything in `release.yml` is slightly less modular than splitting into a second workflow, but it reduces churn and keeps the release story easier to audit in one place.

## Detailed Behavioral Intent

### `cliproxyapi` reliability intent

After this work:

- If `cliproxyapi` crashes or Docker restarts, Docker should still bring it back.
- If repo-managed update/rebuild paths run, they should not casually leave the service absent without a clear recovery path.
- Documentation should stop implying that destructive stack actions are equivalent to lightweight restarts.

This design does **not** attempt to eliminate all downtime windows during force-recreate or full-stack rebuild. It only makes the lifecycle more intentional and recoverable.

### GHCR publish intent

After this work:

- dashboard image publishing to GHCR happens automatically on the chosen release/tag path.
- manual-only release publishing is no longer required for normal registry publication.
- users consuming GHCR get updated dashboard images without needing a separate manual workflow run.

---

## Error Handling and Safety

### Docker lifecycle side

- Prefer deterministic compose-managed recovery where possible.
- If a fallback path must still exist, it should be treated as an explicit degraded recovery path rather than the normal happy path.
- Docs should explicitly warn that `docker compose down` removes containers and is not the same as a continuity-preserving restart.

### GitHub Actions side

- Automatic publish should not silently change registry target or tag naming.
- Trigger changes must avoid accidental duplicate publish behavior between manual and automatic paths.
- Workflow conditions should stay readable enough that future release failures can be diagnosed from the YAML alone.

---

## Verification Criteria

### For `cliproxyapi`

Evidence of success should include:

1. Compose/runtime definition still clearly expresses Docker auto-restart behavior.
2. Repo-managed update path is more recovery-safe and does not rely on fragile removal-first behavior where avoidable.
3. Rebuild/docs distinguish continuity-preserving actions from destructive teardown flows.
4. Relevant scripts remain syntactically valid.

### For GitHub Actions

Evidence of success should include:

1. `release.yml` is no longer manual-only for the intended publish path.
2. The workflow still targets GHCR dashboard image publication.
3. Expected tags are preserved.
4. YAML remains structurally coherent and reviewable.

---

## Testing Strategy

### Docker lifecycle changes

- Add or adjust focused tests for `dashboard/src/app/api/update/route.ts` if the existing route structure supports bounded testing.
- Use shell/script validation for script changes:
  - `bash -n rebuild.sh`
- Use readback/targeted verification for declarative compose/docs changes.

### GitHub Actions changes

- Validate by inspecting trigger, job conditions, image target, and publish steps directly in the workflow file.
- Keep the workflow edits small enough that static review is meaningful.

---

## Risks and Trade-offs

- If the update route still needs a hard fallback, there may remain a small recreate window; this is acceptable within the current scope.
- If workflow automation is broadened too far, it could accidentally change release cadence or duplicate publish steps. Keeping the change small mitigates this.
- Because non-Docker host integrations remain in place by user choice, the system will not become “pure Docker”; this design only improves the Docker-centric parts that matter for the requested behavior.

---

## Final Scope Statement

This work will:

- harden existing Docker/container lifecycle behavior for `cliproxyapi`
- clarify operational behavior around destructive vs continuity-preserving flows
- automate dashboard GHCR publishing on the release/tag path

This work will not:

- remove systemd/cron/host dependencies
- add another registry
- redesign the entire release or deployment architecture
