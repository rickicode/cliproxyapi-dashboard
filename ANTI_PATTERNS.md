# Project Anti-Patterns & Forbidden Patterns

## Explicit Project-Specific Prohibitions

This document lists explicit, project-documented anti-patterns discovered in code, configuration, and documentation. These are NOT generic best practices but rather concrete constraints for this codebase.

---

## 1. SINGLE-INSTANCE CONSTRAINT (Critical Deployment Limitation)

**Pattern:** `providerMutex` AsyncMutex in production multi-instance deployments

**Evidence:**
- File: `dashboard/src/lib/providers/management-api.ts` (lines 4-18)
- Status: Critical blocker

**Constraint:**
```
This AsyncMutex is IN-PROCESS ONLY and does NOT work across multiple dashboard instances.
It only synchronizes concurrent requests within a single Node.js process.

Current deployment: Single-instance (sufficient for current architecture)
If multi-instance deployment is needed in the future: Replace with Postgres advisory locks
or a distributed locking mechanism (Redis Redlock, etc.)
```

**Impact:**
- ❌ **FORBIDDEN**: Using `providerMutex` race-condition protection in multi-instance deployments
- ✅ **Required Migration**: Move to DB-backed advisory locks before horizontal scaling

**Performance Plan Reference:** `docs/PERFORMANCE-RACE-HARDENING-PLAN.md`, Day 5

## 3. IMMUTABLE OBJECT PATTERNS

**Pattern:** Object mutation in state management and transformations

**Evidence:**
- File: `docs/CONTRIBUTING.md` (line 51)
- Requirement: TypeScript + React 19 codebase

**Constraint:**
```
Immutable patterns — create new objects, never mutate
```

**Impact:**
- ❌ **FORBIDDEN**: Direct mutation of objects:
  ```typescript
  // WRONG
  const obj = { ...state };
  obj.field = newValue;
  
  // CORRECT
  const obj = { ...state, field: newValue };
  ```
- ✅ **Required Pattern**: Spread operator, Object.assign, or immutable library updates
- **Scope**: All state management, props transformations, reducer functions

## 4. NO HARDCODED API ENDPOINTS

**Pattern:** String literals for API routes in component/hook code

**Evidence:**
- File: `docs/CONTRIBUTING.md` (line 54)
- PR Checklist: `docs/CONTRIBUTING.md` (line 73)
- Reference Implementation: `lib/api-endpoints.ts`

**Constraint:**
```
Fetch URLs — use `API_ENDPOINTS.*` from `lib/api-endpoints.ts`, no hardcoded strings
```

**Impact:**
- ❌ **FORBIDDEN**: Direct fetch/axios calls with URL strings:
  ```typescript
  // WRONG
  fetch('/api/usage/history?from=2024-01-01')
  
  // CORRECT
  fetch(API_ENDPOINTS.USAGE.HISTORY({ from: '2024-01-01' }))
  ```
- ✅ **Required**: All URLs must be centralized in `API_ENDPOINTS`
- **PR Gate**: Cannot merge without this compliance

## 7. SSH FIREWALL CONSTRAINT

**Pattern:** UFW (Uncomplicated Firewall) enablement without SSH allow-list

**Evidence:**
- File: `infrastructure/docs/ufw-setup.md` (line 54)
- Platform: Server deployments (Linux)

**Constraint:**
```
WARNING: Always allow SSH before enabling UFW, or you may lock yourself out of the server.
```

**Impact:**
- ❌ **FORBIDDEN**: Running `ufw enable` before `ufw allow ssh`
- ✅ **Required Order**:
  ```bash
  ufw default deny incoming
  ufw allow ssh           # FIRST — before enable!
  ufw allow http
  ufw allow https
  ufw enable
  ```
- **Consequence**: Lockout requires physical/console access to recover
